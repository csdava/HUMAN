## 2. 儿童端同事需配合或可对齐的事项

### 2.1 餐图字段 `MealRecord.image_key`（**强烈建议对齐**）

- **现状**：家长端 `templates/parent/dashboard.html` 对今日膳食卡片使用 `<img src="{{ meal.image_key }}">`，并用 `onerror` 隐藏加载失败图。
- **建议约定**：儿童端在上传/保存膳食记录时，将 `image_key` 写成**浏览器可直接请求的完整 URL**（例如走 Django `FileField`/`STATIC`/`MEDIA` 的 URL，或对象存储公网地址）。
- **若暂时只能存相对路径或对象 key**：儿童端同事请在联调说明里写清规则；家长端可再改模板（例如统一前缀 `MEDIA_URL`）— 需双方约定一处，避免各写各的。

### 2.2 绑定码与家长绑定流程（**已存在，勿破坏契约**）

- 儿童注册/账号流程里生成的 **6 位绑定码** 由儿童端展示给家长；**儿童端首页**（`child_dashboard`）顶部常驻展示绑定码与「复制」按钮，若历史数据无码则首次进入首页时由服务端补生成。
- 家长端消费接口：`POST /parent/bind-child/`（需登录），表单字段 `bind_code`。
- **注意**：该接口已加 `@login_required`；未登录只会得到登录跳转页，前端 `fetch` 拿不到 JSON 属正常。

### 2.3 任务待确认数据流（**勿改字段语义**）

- 儿童提交任务 → `TaskRecord` 为 `pending` → 家长端「待确认」列表展示 → `POST /parent/confirm-task/<record_id>/` 确认。
- 若儿童端改任务提交接口、状态枚举或 URL，请同步家长端（`confirmTask` 使用的路径与 CSRF 头）。

### 2.4 鼓励语（**可选增强**）

- 家长端：`POST /parent/send-encouragement/`，儿童端已有「家长鼓励」展示逻辑即可。
- 若儿童端对 `Encouragement.message` 有长度、富文本、敏感词策略，建议文档化；家长端目前发**纯文本**（并会去掉输入中的 HTML 片段）。
- 家长端列表仅展示发给**当前选中孩子**的记录；`Encouragement.is_read` 仍由儿童端「已读」接口维护。

### 2.5 膳食备注 `Child.diet_notes`（**新增，可选对接**）

- 家长在家庭端填写、保存至 `Child.diet_notes`（纯文本，最长约 4000 字）。
- **儿童端同事可选**：在孩子首页或设置页只读展示该字段，避免孩子误改（写权限仅在家长端 `POST /parent/child-diet-notes/`）。
- **`Child.birth_date`（出生日期）与年龄**：**仅家庭端**通过 `POST /parent/child-birth-date/` 维护；年龄由服务端按出生日期推算（`Child.age_years()`）。儿童端、学校端应**只读展示**（用于推荐摄入、周报等），不要提供修改入口。

### 2.6 结构化过敏/医嘱标签 + 过敏预警（**新增：三端共享**）

- 结构化字段：
  - `Child.allergy_tags`: `list[str]`，例如 `["花生","牛奶"]`
  - `Child.medical_tags`: `list[str]`，例如 `["少油","控糖"]`
- 家长端写入接口：`POST /parent/child-health-tags/`（表单字段为 JSON 字符串）：
  - `allergy_tags`: JSON array 字符串，例如 `["花生","牛奶"]`
  - `medical_tags`: JSON array 字符串，例如 `["少油"]`
- 过敏命中检测：
  - 后端在儿童端上传识别接口 `POST /api/yolo/recognize/` 识别出食材后，会用 “标签是否为食材名子串” 做命中（如 “花生” 命中 “花生米”）。命中则：
    1) 写入 `HealthAlert(alert_type='allergy')`
    2) 在响应里返回 `allergy_hits`，前端可即时弹窗提示。
  - 家长端 `POST /parent/correct-meal/<meal_id>/` 修正后也会再次检测并写入预警（防漏检）。
- 预警拉取与已读接口（供三端 UI 轮询/展示）：
  - 儿童端：`GET /child/alerts/`、`POST /child/alerts/<id>/read/`；首页已内嵌预警列表（与家庭端列表同源）。
  - **参考摄入 + 标签（JSON，与家庭端口径一致）**：`GET /child/recommended-intake/`（儿童登录）；`GET /parent/recommended-intake/`（家长登录）二者均返回 `recommendation`、`allergy_tags`、`medical_tags`。
  - 家长端：`GET /parent/alerts/`、`POST /parent/alerts/<id>/read/`
  - 学校端：`GET /school/alerts/`、`POST /school/alerts/<id>/read/`（仅本班级孩子）
- 学校端 `school_dashboard` →「班级名册」卡片中展示每名学生的参考摄入区间与过敏/医嘱标签（**与家长端、儿童端只读展示同源**）。

### 2.7 五色宝石 `Child.gem_*`（**与食谱推荐联动**）

- 家长端食谱接口：`GET /parent/recipes/?fill_gaps=1` 会读当前选中孩子的 `gem_red` … `gem_purple`，在食谱库中做多关键词检索，**面向全库**（系统食谱与家长自建食谱一并检索），**不是**「只推我家刚加的食谱」。
- **排序**（`fill_gaps=1` 且已选中孩子时）：先按 `gap_match_score` 降序（单条食谱能补上的「当日未点亮」色越多分越高），同分下 **`created_by` 为空的系统食谱优先**，再按名称排序；响应里每条带 `gap_match_score`、`is_family_recipe`。
- **儿童端若调整**：「何时写 gem」「是否按天重置」「多餐累计规则」变化时，请通知家长端同事评估 `fill_gaps` 推荐是否仍合理（逻辑在 `core/views.py` → `parent_recipes`）。

### 2.8 家长修正膳食后的数据（**已落地**）

- `POST /parent/correct-meal/<meal_id>/` 会重写 `MealFoodItem`，重算营养与 **`MealRecord.total_score`**（按本餐食材覆盖的五色大类数 1–5），并置 `is_verified=True`。
- 儿童端若展示「单日总分、五色是否全」等聚合逻辑，请确认是否与「单餐 `total_score`」展示一致，避免用户理解冲突（产品层可一起定文案）。

## 3. 学校端同事需配合或可对齐的事项

### 3.1 教师与班级列表（**依赖学校端维护的数据**）

- 家长端：`GET /parent/get-teachers/` → JSON `teachers[].id, name, school, class_name`（`school` 已为学校名称字符串）。
- 进班 / **换班**：`POST /parent/child-class/`（表单 `child_id` 必填；`teacher_id` 有值则先清空该孩子全部进班再进入新班；`teacher_id` 留空则仅退出全部班级）。兼容旧接口 `POST /parent/add-to-class/`（仅追加同教师班级，不自动退班）。
- **若学校端**改 `Teacher` / `School` 字段或改接口权限，需同步家长端下拉与文案。

### 3.2 管理员 / 后台食谱数据（**非学校端独占，但常由运营录入**）

- 家长端食谱来自模型 `Recipe`；若库中无数据，页面会提示「暂无食谱数据」。
- 负责录入的同事（或学校端若管食谱）需保证 `target_nutrients`、`ingredients`、`name` 等含常见中文关键词，以便 `fill_gaps` 检索命中。

## 4. 家长端新增/强依赖的 HTTP 接口（给联调清单）

以下均需**家长账号登录**（Session），并带 CSRF（POST）。

| 方法 | 路径 | 说明 |
|------|------|------|
| GET  | `/parent/meal/<meal_id>/detail/` | 单条膳食 JSON（含 `foods[]`：`food_id`, `name`, `weight`, `category`） |
| GET  | `/parent/food-materials/` | 全量 `FoodMaterial` 下拉用列表 |
| POST | `/parent/correct-meal/<meal_id>/` | 表单 `food_ids`（多值）、`weights`（多值），与 `food_ids` 顺序一一对应；可为空表示清空本餐食材 |
| GET  | `/parent/meal-report/?days=7` | `days` 建议 1–90；统计区间**含今天**共 `days` 个自然日 |
| GET  | `/parent/meal-list/?days=14` | `days` 默认 14、最大 60；按日期分组返回近期每餐摘要，供历史列表与「修正」入口 |
| POST | `/parent/child-diet-notes/` | 表单 `diet_notes` |
| POST | `/parent/child-birth-date/` | **仅家庭端**：表单 `birth_date`（`YYYY-MM-DD`）或空字符串清空；返回 `age_years` |
| POST | `/parent/child-health-tags/` | 表单 JSON 字符串字段 `allergy_tags`、`medical_tags` |
| GET  | `/parent/recommended-intake/` | 返回当前孩子 `age_years`、`recommendation`（热量/蛋白区间与说明）、`allergy_tags`、`medical_tags`（与儿童端 `GET /child/recommended-intake/` 字段一致） |
| GET  | `/child/recommended-intake/` | 儿童登录：同上，供 App/页面拉 JSON；首页亦服务端渲染参考摄入卡片 |
| GET  | `/parent/alerts/` | 预警列表 |
| POST | `/parent/alerts/<id>/read/` | 家长端标记预警已读 |
| GET  | `/parent/export-weekly-pdf/?days=7` | 下载 PDF 周报；`days` 默认 7、最大 30 |
| GET  | `/parent/recipes/` | 可选 `fill_gaps=1`、`target=`、`limit=`（默认 30，最大 80）。`fill_gaps=1` 时排序与缺口分见 **§2.7**；每条食谱含 `gap_match_score`（有选中孩子时）。 |
| POST | `/parent/recipes/add/` | **家长账号**：`name`、`ingredients` 必填；可选 `description`、`steps`、`target_nutrients`、`suitable_for`。**热量/蛋白质/碳水/脂肪由服务端根据 `ingredients` 自动估算**（`core/recipe_nutrition_estimate.py`：优先 `FoodMaterial`，否则内置常见食材表）；请求里若传 `calories` 等宏量字段也会被忽略。若未传 `target_nutrients`，会用估算结果里的 `suggested_target_nutrients` 写入。成功响应含 `nutrition_estimate`（含 `lines[]` 逐行匹配说明）。 |
| POST | `/parent/confirm-task/<record_id>/` | 成功时额外返回：`new_power`, `new_level`, `power_to_next`, `today_completed`, `week_completed`, `pending_count`, `progress_percent`, `upgraded` |
| POST | `/parent/add-manual-task/` | 成功时同样返回上述统计字段（与确认任务一致，便于前端无刷新更新） |
| POST | `/parent/child-class/` | **换班 / 退出班级**：`child_id` 必填；`teacher_id` 为教师 id 时先删除该孩子所有 `ClassStudent` 再进入所选班级；`teacher_id` 留空或 `0` 则仅退出全部班级。成功返回 `enrollments`（当前列表，换班后通常一条） |
| POST | `/parent/add-to-class/` | 旧接口：将孩子加入某教师班级；若该 `(teacher, child)` 已存在会失败；换班请用 `child-class` |
